# v1.3.0 Sprint 1 Plan — Connection Lifecycle UX

## Goal

Land the first meaningful slice of **Desktop Beta Quality** by making the connection lifecycle understandable, predictable, and harder to misuse.

This sprint is the execution plan for **Epic B — Connection lifecycle UX** from `docs/v1.3.0-backlog.md`.

---

## Sprint outcome

After this sprint:

- users can clearly see whether the app is `idle`, `connecting`, `connected`, `disconnecting`, or `error`
- active profile is more visible in the main product surface
- connect/disconnect actions are guarded during in-flight transitions
- common failure states expose actionable recovery paths
- runtime exit reasons are mapped into user-facing language instead of raw internal signals

---

## Scope

### In scope
- connection lifecycle state model in product/UI layer
- dashboard/profile surface updates for active profile visibility
- CTA behavior during in-flight transitions
- failure-state recovery actions
- user-facing runtime/session error mapping
- widget/application tests for key lifecycle callouts

### Out of scope
- tray/menu integration
- diagnostics export redesign
- crash persistence / stale session recovery
- updater / release metadata work

---

## Work items

### 1. Define lifecycle state contract

- [x] audit current status model(s) in controller + UI layer
- [x] define a single user-facing lifecycle vocabulary:
  - `idle`
  - `connecting`
  - `connected`
  - `disconnecting`
  - `error`
- [x] decide whether this belongs in a dedicated presentation/view-model type or can be derived cleanly from existing controller state
- [x] document ambiguous mappings explicitly (for example: runtime exited cleanly while UI still says disconnecting)

**Target files (likely)**
- `client/lib/features/controller/application/adapter_backed_client_controller.dart`
- `client/lib/features/controller/application/client_controller_api.dart`
- `client/lib/features/dashboard/presentation/dashboard_page.dart`

---

### 2. Improve Dashboard lifecycle presentation

- [x] make current connection state more visually explicit on Dashboard
- [x] show active profile name/id/status in the primary status area
- [x] add state-specific helper copy
  - connecting → “正在建立连接...”
  - disconnecting → “正在断开连接...”
  - error → actionable summary + next step
- [x] prevent status card from mixing stale active profile info with disconnected/error states

**Target files (likely)**
- `client/lib/features/dashboard/presentation/dashboard_page.dart`
- shared widgets used by dashboard status sections

---

### 3. Guard connect/disconnect actions

- [x] disable `Connect` while already connecting/connected
- [x] disable `Disconnect` while already disconnecting/idle
- [x] prevent double-submit / rapid-tap state corruption
- [x] ensure button labels reflect current transition state when practical

**Target files (likely)**
- `client/lib/features/dashboard/presentation/dashboard_page.dart`
- possibly profile action surfaces if connect can also be triggered there

---

### 4. Add recovery actions for failure states

- [x] define first-pass recovery CTA set:
  - `Retry`
  - `Open Profiles`
  - `Open Troubleshooting` (or placeholder help surface if no full page exists yet)
- [x] wire CTA visibility by error category / state
- [x] avoid showing useless actions for non-recoverable/internal states

**Target files (likely)**
- `client/lib/features/dashboard/presentation/dashboard_page.dart`
- optional supporting presentation helpers

---

### 5. Map runtime exit/failure reasons to user-facing messages

- [x] inventory current runtime/session failure reasons exposed by controller/adapter
- [x] define a translation layer from internal signals to human-readable product messages
- [x] distinguish at least:
  - failed launch / config generation issue
  - runtime exited unexpectedly
  - user-triggered disconnect
  - profile/config input issue
- [x] ensure raw/internal details are still available for diagnostics, but not as the primary UX string

**Target files (likely)**
- `client/lib/features/controller/application/adapter_backed_client_controller.dart`
- `client/lib/features/controller/application/real_shell_controller_adapter.dart`
- dashboard presentation layer

---

### 6. Add tests

- [x] application tests for lifecycle state mapping
- [x] widget tests for dashboard callouts and button enable/disable rules
- [x] tests for recovery CTA visibility in representative failure states
- [x] tests for active profile visibility during connect/error/disconnect transitions

**Target files (likely)**
- `client/test/features/controller/application/adapter_backed_client_controller_test.dart`
- new/updated dashboard widget tests

## Suggested delivery order inside the sprint

1. lifecycle contract
2. runtime/error message mapping
3. dashboard presentation updates
4. guarded actions
5. recovery CTAs
6. tests and polish

---

## Definition of done

Sprint 1 is done only when:

- lifecycle states are visually and semantically clearer than the current MVP shell
- connect/disconnect actions cannot be spammed into nonsense states
- at least the primary failure cases present understandable recovery options
- tests cover the key lifecycle transitions and UI gating logic

If the result is only “nicer labels” without state/CTA correctness, the sprint is not done.
