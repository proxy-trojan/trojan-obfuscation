## G. Commit packaging checklist

Before committing:

- [ ] review `git diff --stat`
- [ ] make sure no unrelated generated junk is included
- [ ] confirm tests/docs were updated in the same change set
- [ ] decide whether this should be one commit or two commits:
  - lifecycle/support behavior
  - release docs / handoff notes

Suggested commit split:

### Commit 1 — product behavior

- dashboard lifecycle/external-activation UX
- shell auto-home behavior
- desktop lifecycle models/service changes
- diagnostics/support UX and persistence changes
- corresponding tests

Suggested message:

```bash
git commit -m "feat(client): polish desktop lifecycle and support surfaces"
```

### Commit 2 — release posture docs

- beta handoff notes
- Linux smoke report refresh
- staging release checklist
- docs index updates if needed

Suggested message:

```bash
git commit -m "docs(client): add v1.3.0 beta handoff and staging checklist"
```

If you prefer one squashed commit for the candidate, at least keep the release notes/docs inside the same commit so the branch remains self-describing.
