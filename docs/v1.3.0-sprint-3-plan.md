# v1.3.0 Sprint 3 Plan — Crash Handling and Recovery Polish

## Goal

Make the desktop client recoverable and understandable after abnormal runtime/app states.

This sprint executes **Epic D — Crash handling and recovery polish** from `docs/v1.3.0-backlog.md`.

---

## Sprint outcome

After this sprint:

- the app preserves a useful summary of the last runtime failure
- stale session state is recognized and recovered more safely on next launch
- generated config/temp artifacts are cleaned up more consistently after abnormal exits
- failure and restart paths have explicit regression coverage

---

## Scope

### In scope
- last runtime failure summary capture/persistence
- stale session recovery on next launch
- cleanup guarantees for generated config/temp artifacts after abnormal exits
- restart-after-failure regression tests

### Out of scope
- tray/menu features
- broader diagnostics redesign
- updater work
- public beta release process changes

---

## Work items

### 1. Persist last runtime failure summary
- [x] define a persisted state model for latest runtime failure
- [x] record failure summary when runtime exits unexpectedly or launch fails
- [x] surface the summary to product/support surfaces

### 2. Add stale session recovery
- [x] detect stale “was running / active profile” state on next launch
- [x] downgrade to a safe recovered state instead of pretending the session is still healthy
- [x] expose a user-facing recovery note when this happens

### 3. Improve cleanup guarantees
- [x] audit generated runtime config/temp file cleanup paths
- [x] ensure abnormal stop / failed launch does not leave misleading artifacts behind
- [x] add tests for cleanup after failure/restart

### 4. Add regression coverage
- [x] failed-launch summary persistence tests
- [x] runtime-exit summary persistence tests
- [x] stale session recovery tests
- [x] restart-after-failure tests

---

## Suggested delivery order

1. last runtime failure summary
2. stale session recovery
3. cleanup guarantees
4. regression tests and polish

---

## Implementation notes

Delivered behavior in this sprint:

- controller persists both a `lastRuntimeFailureSummary` and a `runtimeSessionSnapshot`
- bootstrap restores persisted state on launch and downgrades stale runtime state into a truthful recovery error
- recovery pass performs best-effort cleanup for managed runtime config artifacts
- real shell adapter no longer pretends disconnect finished before process exit actually happens
- retry-after-failure and disconnect-after-runtime-noise regressions are covered explicitly in controller tests

## Definition of done

Sprint 3 is done only when:

- a runtime/app failure leaves behind a useful summary instead of a confusing blank state
- stale session state is recovered into something truthful and supportable
- cleanup behavior is consistent enough that retries do not inherit misleading residue
- failure/restart paths are covered by explicit tests
