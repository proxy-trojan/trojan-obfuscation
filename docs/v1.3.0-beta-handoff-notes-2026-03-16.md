# v1.3.0 Beta Handoff Notes — 2026-03-16

## Recommended posture

This candidate is suitable for:

- **Linux staging handoff**
- **small desktop beta** with informed testers
- validating desktop lifecycle, supportability, and diagnostics UX

This candidate is **not yet positioned** for:

- broad public desktop beta promotion
- strong tray-behavior claims across Linux environments
- claiming that real runtime / packaging maturity is equally validated on every desktop platform

---

## What improved in this candidate

### 1. Desktop lifecycle polish is now user-visible

The desktop shell now behaves more like a product and less like an internal tool:

- close / minimize / quit semantics are exposed in Settings
- duplicate-launch mitigation is implemented through a single-instance guard
- a second launch now hands focus back to the existing window instead of silently creating ambiguity
- external activation is surfaced in-product rather than hidden in logs

### 2. Recent desktop activation flow is now a full UX chain

When a secondary desktop launch focuses the existing window:

- the primary instance records the activation
- the shell automatically returns to **Home**
- the dashboard shows a **Recent desktop activation** card
- the card explains *why* the window came back
- the card exposes lightweight lifecycle facts:
  - single-instance guard status
  - tray integration readiness
  - close interception status
- the card offers:
  - **Review desktop behavior** → jumps to Settings
  - **Dismiss / Got it**
- the reminder also **ages out automatically** to avoid stale noise

### 3. Diagnostics / support flow is more truthful

This candidate also includes the supportability work landed alongside Sprint 4 polish:

- app-level uncaught runtime error summaries are persisted and surfaced
- diagnostics export is routed through the support-bundle path
- diagnostics export failures now show categorized guidance instead of raw failure only
- support/problem-report surfaces are more explicit about runtime state and failure context

## Honest known limitations for testers

### 1. Linux tray behavior is environment-dependent

If Linux `appindicator` support is missing:

- build still succeeds
- the desktop client still launches as a normal app candidate
- tray integration falls back to **unavailable / no-op**

This is a **known limitation**, not a packaging failure.

### 2. Current Linux smoke evidence is not a full desktop GUI sign-off

On 2026-03-16:

- headless validation captured useful packaging and duplicate-launch evidence
- but it did **not** provide a complete GUI/tray/window end-to-end sign-off

So this candidate should be treated as:

- **validated enough for staging / small beta**
- **not yet a fully confidence-maxed public Linux beta**

### 3. Updater remains boundary-only

- update channel and update-related product surface exist
- this should still be treated as **skeleton / contract UI**, not a production updater claim

### 4. Real runtime validation is narrower than UI validation

- widget / unit validation is strong
- release-bundle runtime evidence is improving
- one real desktop-environment smoke pass is still recommended before wider promotion

## Suggested tester focus

Ask testers to focus on these paths:

### Desktop lifecycle

- launch the app normally
- launch it a second time and verify focus handoff behavior
- close / minimize / reopen / quit behavior matches Settings
- confirm whether tray is available in their Linux desktop environment

### Profile + connection basics

- create or import a profile
- store Trojan password
- connect once
- disconnect once
- check whether lifecycle copy stays understandable throughout

### Supportability

- open Advanced / Troubleshooting
- export diagnostics bundle
- confirm support copy is clear when runtime is healthy and when it fails

### Linux environment notes worth capturing

Please ask testers to include:

- distro + version
- desktop environment / window manager
- whether appindicator is installed
- whether tray icon is visible
- whether second launch restores focus correctly
- whether diagnostics export writes successfully to disk

## Suggested release wording

Recommended wording for this handoff:

> `v1.3.0` is currently a **desktop-beta staging candidate** focused on lifecycle clarity, diagnostics/supportability, and desktop behavior predictability. Linux packaging is buildable and test-green; tray behavior may degrade to a no-op fallback in environments without appindicator support.

Avoid wording like:

- “production-ready desktop client”
- “fully validated multi-platform tray experience”
- “public beta ready on all Linux desktop environments”

## Minimum next step before wider promotion

Run one real desktop Linux smoke pass against the release bundle and verify:

1. launch succeeds in a normal desktop session
2. duplicate launch focuses existing window
3. tray behavior matches platform capability
4. Settings lifecycle copy matches actual behavior
5. diagnostics export succeeds on a real file-backed path

Until that is done, the honest posture remains:

- **ship as staging / small beta**
- **do not overclaim broader desktop readiness**
