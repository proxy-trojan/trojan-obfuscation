# v1.3.0 Backlog — Desktop Beta Quality

## Goal

Make the desktop client feel like a real product instead of a packaged engineering shell.

This milestone should improve the **connection lifecycle UX**, **desktop behavior**, **error recovery**, and **supportability** without expanding scope into public-beta distribution or speculative protocol work.

---

## Scope guardrails

### In scope
- tray/menu behavior
- stronger connection lifecycle UX
- clearer error and recovery UX
- more dependable diagnostics/support flow
- update channel skeleton
- crash / failure handling polish
- repeatable desktop beta validation

### Explicitly out of scope
- public beta onboarding / broad external distribution
- code-signing / notarization hardening
- major protocol expansion / new protocol families
- mobile-first UX work
- deep trusted-front / public-edge capability expansion

---

## Epic A — Desktop lifecycle polish

### Objective
Make the desktop app behave like a native desktop product during day-to-day use.

- [ ] Add tray/menu bar behavior for desktop platforms
- [ ] Define close / minimize / quit semantics clearly
- [ ] Add quick actions from tray/menu (`Connect`, `Disconnect`, `Open`, `Quit`)
- [ ] Show selected profile + current status in tray/menu label when practical
- [ ] Prevent confusing duplicate windows / duplicate app launches
- [ ] Add explicit app lifecycle handling for reconnect-safe resume behavior

### Exit criteria
- user can leave the app running without confusion
- app close behavior is predictable
- basic connect/disconnect control is available without reopening the full window

---

## Epic B — Connection lifecycle UX

### Objective
Turn the current controller-backed flow into a clearer user-facing connection experience.

- [ ] Add clearer lifecycle states in UI (`idle`, `connecting`, `connected`, `disconnecting`, `error`)
- [ ] Improve progress copy for connect / disconnect / failure transitions
- [ ] Surface the currently active profile more clearly across Dashboard / Profiles
- [ ] Make connect/disconnect actions harder to misfire during in-flight state transitions
- [ ] Add recovery actions for common failure states (`Retry`, `Open Troubleshooting`, `Open Profiles`)
- [ ] Ensure runtime session end reasons are mapped to understandable user messages
- [ ] Add widget tests for primary lifecycle callouts and recovery actions

### Exit criteria
- first-run user can understand what the app is doing during one connection attempt
- failure states present clear next actions instead of raw engineering signals

---

## Epic C — Diagnostics and support UX

### Objective
Make the client supportable enough for beta testing without requiring internal context.

- [ ] Make diagnostics export entrypoint more obvious in the UI
- [ ] Add a short support bundle summary before export (what is included / excluded)
- [ ] Distinguish user-facing import/config errors from runtime/OS/process errors consistently
- [ ] Add structured troubleshooting cards for the top recurring failure categories
- [ ] Improve log/session rendering readability (timestamps, severity, grouping, truncation rules)
- [ ] Validate diagnostics export on real file-backed paths across desktop flows
- [ ] Add regression tests for diagnostics/export failure messaging

### Exit criteria
- tester can produce a useful support bundle without developer guidance
- exported diagnostics are predictable and explainable

---

## Epic D — Crash handling and recovery polish

### Objective
Reduce the “engineering shell” feel when the runtime or app hits abnormal states.

- [ ] Add app-level error boundary / guarded async reporting for critical flows
- [ ] Capture and present last runtime failure context more consistently
- [ ] Persist last known runtime/session summary for post-failure inspection
- [ ] Improve cleanup guarantees for generated config / temp files after abnormal exits
- [ ] Add explicit stale-session recovery on next launch
- [ ] Add tests for abnormal runtime stop / failed-launch cleanup / restart-after-failure

### Exit criteria
- runtime failures do not leave the app in a confusing half-state
- support/debug flow after a failure is clearer than “try again and hope”

---

## Epic E — Update channel skeleton

### Objective
Create the product boundary for future self-update work without overcommitting implementation scope.

- [ ] Define update channel model in settings (`stable`, `beta`, optional `internal`)
- [ ] Show current channel and update policy in Settings
- [ ] Add placeholder update-check action with clear “not yet wired” / stub semantics if backend is not ready
- [ ] Define release metadata contract needed by a future updater
- [ ] Document what v1.3.0 does **not** promise about self-update yet

### Exit criteria
- update channel is visible in product model and docs
- future updater work has a stable contract boundary

---

## Epic F — Desktop beta validation

### Objective
Strengthen pre-release confidence for desktop beta candidates.

- [ ] Add desktop beta checklist derived from real install/start/connect/export workflows
- [ ] Add a documented manual validation matrix for Linux / Windows / macOS
- [ ] Add one higher-signal runtime smoke pass for a real connect/disconnect lifecycle where feasible
- [ ] Validate headless vs GUI smoke expectations explicitly in docs/runbooks
- [ ] Ensure release notes / known limitations template exists for beta candidates

### Exit criteria
- desktop beta candidates can be tested with low ambiguity
- release validation is understandable to someone other than the author

---

## Suggested delivery order

1. **Epic B — Connection lifecycle UX**
2. **Epic C — Diagnostics and support UX**
3. **Epic D — Crash handling and recovery polish**
4. **Epic A — Desktop lifecycle polish**
5. **Epic E — Update channel skeleton**
6. **Epic F — Desktop beta validation**

Reasoning:
- lifecycle clarity and troubleshooting quality will improve real usability faster than tray polish
- crash/recovery quality matters before trying to present the client as a beta product
- update channel should exist as a boundary, but not steal focus from core beta usability

---

## Milestone exit verdict

`v1.3.0` should be considered complete only when:

- desktop lifecycle feels predictable
- connect/disconnect/failure states are understandable
- diagnostics/support flow is dependable
- desktop beta validation is repeatable without internal tribal knowledge

If those are not true, the milestone is not done — even if packaging/builds remain green.
