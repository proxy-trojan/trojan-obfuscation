# v1.5.0-beta.3 Platform Dataplane Integration 设计规格（Hard Gate A）

## 1. 背景

`v1.5.0-beta.2` 已完成：
- routing kernel 与规则编辑基线
- routing safety（preflight / dry-run / rollback / quarantine）
- diagnostics recovery evidence 与对应 CI gate

但当前下一阶段的主矛盾不再是“功能是否存在”，而是“**三平台是否都能提供 runtime-true 的 dataplane 证据**”。

用户已确认采用：
- 双线并行推进（主线 + 并行线）
- 主里程碑为平台 dataplane integration
- 放行标准采用 **A（硬标准）**

---

## 2. 目标与范围

## 2.1 主目标（主里程碑）

在 `v1.5.0-beta.3` 前建立并验证：

1. Linux / Windows / macOS 三平台 dataplane probe 均走 runtime-true 路径。
2. 证据链统一可导出、可比对、可回归（runner → verdict → diagnostics/export）。
3. CI 与 release gate 对 runtime-true 形成硬门禁。

## 2.2 并行目标（次主线）

在不阻断主里程碑的前提下，继续推进 rule/policy CRUD 完整化与 UX 收敛：
- profile editor / details / import-export 的一致性与可维护性增强
- guardrail 与可解释反馈统一

## 2.3 非目标（本阶段不做）

- 远程 rule-set 分发平台化（server 侧治理）
- 自动更新 feed 完整实现
- 移动端能力追平桌面端

---

## 3. 放行标准（A：硬标准）

`v1.5.0-beta.3` 仅在以下条件全部满足时可发布：

1. **三平台 runtime-true dataplane 证据齐备**：
   - Linux: runtime-true
   - Windows: runtime-true
   - macOS: runtime-true
2. 任一平台若落入 fallback/stub posture，则视为**未达标**，阻断 beta.3 发布。
3. release notes 不得以“主能力已完成”口径掩盖 fallback 状态。

---

## 4. 设计方案

## 4.1 证据主干（统一模型）

沿用现有 routing testing 子系统，统一三平台 evidence 输出语义：

- `scenario`（输入条件与期望）
- `observation`（真实执行结果）
- `verdict`（pass/fail/not_applicable）
- `runtime posture`（real / fallback / stub）
- `adapter identity`（platform + adapter kind + capability）

原则：**模型先统一，再扩平台实现**；避免平台特例把证据语义撕裂。

## 4.2 平台执行层（Windows / macOS 补齐）

在现有 Linux 可执行路径基础上，将 Windows / macOS probe adapter 从 fallback 能力提升为 runtime-true：

- 明确 adapter capability contract（可执行、可观测、可归因）
- 禁止“仅返回 deterministic identity”的伪执行通过主门禁
- probe 失败应归因为可诊断 failure family，而不是静默降级

## 4.3 CI / Release Gate 升级

在现有 targeted gate 之上增加 runtime-truth 约束：

1. Targeted tests 必须覆盖三平台 runtime posture 断言。
2. Packaged smoke 必须验证 evidence payload 中 runtime posture 字段。
3. Validate-artifacts 阶段需明确主能力是否为 runtime-true，而非仅校验产物存在。

## 4.4 Support/Diagnostics 对齐

保持“产品反馈、导出证据、支持口径”三者一致：

- Diagnostics 导出可区分 real/fallback/stub 与对应原因
- Profiles/Diagnostics 页面不把 fallback 描述成已达 runtime-true
- support 文案优先表达“发生了什么 + 下一步做什么”

---

## 5. 并行线（Rule/Policy CRUD）边界

并行线可持续推进，但遵守以下约束：

1. 不能改变主里程碑的放行定义（不得把 CRUD 完整化替代 runtime-true 门禁）。
2. 新增/重构的 CRUD 逻辑必须继承现有 guardrail / dry-run / diagnostics 语义。
3. 若并行线引入风险，优先保证主里程碑链路稳定。

---

## 6. 验收条件

## 6.1 主里程碑验收（必须）

1. 三平台 probe runner 均能输出 runtime-true evidence。
2. routing evidence / recovery evidence 在 diagnostics export 中结构完整。
3. CI smoke + targeted gate + release workflow 全绿且不含 fallback 伪通过。
4. release notes 与 `docs/v1.5.0-release-gates.md` 口径一致。

## 6.2 并行线验收（可分批）

1. rule/policy CRUD 的编辑、校验、导入导出行为一致。
2. UI 提示、guardrail 阻断、diagnostics 反馈语义一致。
3. 新增能力不降低既有 routing safety 门禁。

---

## 7. 风险与缓解

1. **平台环境差异导致 false negative**
   - 缓解：将环境依赖缺失与功能缺失分开归因，避免误导结论。
2. **fallback 被误判为“可发布”**
   - 缓解：将 runtime posture 纳入硬 gate 字段与断言。
3. **并行线干扰主线节奏**
   - 缓解：以主里程碑任务板为第一优先级，CRUD 以可中断小批次交付。

---

## 8. 交付策略

- 采用双线并行：
  - 主线：platform dataplane integration（beta.3 放行基准）
  - 并行线：rule/policy CRUD 完整化（不中断主线）
- 每个批次必须带验证证据，不以“代码已写完”替代“能力已闭环”。

---

## 9. 结论

本规格将 `v1.5.0-beta.3` 的成功定义明确为：

> **三平台 runtime-true dataplane 证据闭环已成立。**

在此之前，不进入“主能力完成”的发布叙事。
