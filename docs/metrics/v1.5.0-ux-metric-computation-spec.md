# v1.5.0 UX Metric Computation Spec

本文档定义 FCSR/HFE/SSR/STE 的计算口径、窗口与过滤规则。文档冻结后，本轮优化期间不得中途改写统计语义。

## 1. FCSR（First-Connect Success Rate）

### 1.1 定义
新用户在首次会话窗口内完成 runtime-true ready 的比例。

### 1.2 计算
- `numerator`：满足以下条件的去重 `user_id` 数量
  1) 发生 `runtime_session_ready_runtime_true`
  2) 该事件发生在对应 `first_session_started` 后 30 分钟窗口内
- `denominator`：发生 `first_session_started` 的去重 `user_id` 数量
- `fcsr = numerator / denominator`（denominator 为 0 时记为 0）

### 1.3 过滤规则
- 排除 `runtime_posture in {fallback, stub}` 的 ready 信号
- 排除缺失 `user_id` 或 `event_at` 的脏数据
- 同一用户多次 ready 只计一次

## 2. HFE（High-Frequency Efficiency）

### 2.1 定义
高频动作（connect/disconnect/switch）的效率指数，反映“完成快、步骤少、返工低”。

### 2.2 原子指标
- `T_action`：从动作意图事件（`quick_connect_clicked` / `quick_disconnect_clicked` / `profile_switched`）到对应 `action_completed` 的中位耗时（秒）
- `N_steps`：`action_completed.interaction_steps` 的中位数
- `R_rework`：`action_rework_detected` / `action_completed` 比率

### 2.3 归一化与加权
对三个原子指标做同口径归一化后计算：

`HFE_index = 0.5 * T_action_norm + 0.3 * N_steps_norm + 0.2 * R_rework_norm`

说明：
- 归一化区间使用同一批次 baseline/candidate 联合区间，避免版本漂移。
- 指标方向统一为“值越小效率越高”；对外可额外输出 `HFE_score = 1 - HFE_index`。

## 3. SSR（Self-Service Recovery）

### 3.1 定义
用户在无需人工支持介入情况下完成异常恢复的比例。

### 3.2 计算
- `numerator`：发生 `recovery_outcome` 且 `outcome == success` 的去重 `session_id`
- `denominator`：发生 `recovery_suggested` 的去重 `session_id`
- `ssr = numerator / denominator`（denominator 为 0 时记为 0）

### 3.3 acted/outcome 闭环护栏
除 SSR 外，Iter-2 需要附加闭环护栏：
- `recovery_acted_rate = count(recovery_action_executed) / count(recovery_suggested)`
- `recovery_outcome_breakdown = {success, fail, abandon}`（按事件数统计）

该闭环用于回答“系统有建议，用户是否执行，执行后是否有效”。

## 4. STE（Support Triage Efficiency）

### 4.1 定义
诊断导出对支持分诊的有效性比例（是否具备必要证据字段）。

### 4.2 计算
- `numerator`：`diagnostics_export_completed` 且满足：
  - 含 `runtime_posture`
  - 含 recovery 相关证据（允许空值但字段必须存在）
- `denominator`：`diagnostics_export_completed` 总数
- `ste = numerator / denominator`（denominator 为 0 时记为 0）

## 5. 窗口与去重规则

1. 时间戳统一使用 UTC。
2. FCSR 按 `user_id` 去重；SSR 按 `session_id` 去重；HFE/STE 按事件级统计后聚合。
3. 跨天会话按 `first_session_started` 所在日期归档。
4. 迟到事件（晚到 >24h）默认不纳入当日实时看板，但可在离线重算任务中修正。

## 6. 对比与收口判定

建议收口阈值（相对 baseline）：
- `FCSR` 提升 ≥ 20%
- `HFE` 关键路径效率提升 ≥ 30%（以 `HFE_score` 或原子指标共同判定）
- `SSR/STE` 不退化，且至少一项提升

同时满足连续两个候选窗口，判定本轮优化可收口。

## 7. Daily-action performance baseline（Iter-3 perf guardrails）

本节定义“高频动作性能基线”的证据与回归护栏口径。其目标是：在不依赖真实远端环境的前提下，让连接/切换等日常路径的性能退化能在 PR/CI 中被显式暴露。

### 7.1 Baseline schema
- 文件：`scripts/perf/compute_daily_action_perf_baseline.py`
- schema：`PerfBaseline(schema_version=1, profile, samples[])`
- 当前采样动作：`connect_preflight`
  - `connect_ready_ms`：进程启动到本地 SOCKS 端口开始监听的耗时（代理“connect ready”）
  - `cpu_time_ms`：采样时点的进程 CPU time（Linux /proc best-effort）
  - `rss_kb`：采样时点的进程 RSS（Linux /proc best-effort）

### 7.2 Regression policy (default)
- `max_ready_ms_regression_ratio = 1.3`
- `max_cpu_time_regression_ratio = 1.5`
- `max_rss_regression_ratio = 1.5`

说明：初始版本只做 fail-closed 的最小护栏；未来若需要 warn/soft gate，可在 policy 上扩展而不破坏 schema。

### 7.3 Evidence / CI gate
- CI 使用 synthetic fixtures 验证 schema + evaluation determinism：
  - `python3 -m pytest scripts/tests/test_compute_daily_action_perf_baseline.py -q`
  - fixtures 位于 `scripts/tests/fixtures/*.json`
- human-run（可选）：本地可通过 `collect` 子命令生成 baseline artifact。
