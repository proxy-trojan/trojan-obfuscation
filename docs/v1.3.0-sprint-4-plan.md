# v1.3.0 Sprint 4 Plan — Desktop Lifecycle Polish

## Goal

Make the desktop client behave more like a native desktop product during day-to-day use.

This sprint executes **Epic A — Desktop lifecycle polish** from `docs/v1.3.0-backlog.md`.

---

## Why this sprint now

Sprint 1–3 established the core product loop:

- connection lifecycle UX is understandable
- diagnostics/support flow is usable
- crash/failure/recovery behavior is more truthful and supportable

The next highest-value gap is no longer recovery correctness — it is **desktop behavior predictability**.

Today the app is still a single-window shell with no tray/menu behavior, no explicit close/minimize semantics, and no protection against confusing duplicate app/window behavior. That is exactly the sort of thing that makes a beta feel like an internal tool.

---

## Sprint outcome

After this sprint:

- desktop close / minimize / quit behavior is explicit and predictable
- user can keep the app running without wondering whether it is still active
- at least one lightweight desktop control surface exists outside the main window
- duplicate launch/window behavior is less confusing
- lifecycle behavior is covered by targeted desktop-facing tests and docs

---

## Scope

### In scope
- desktop lifecycle policy (`close`, `minimize`, `quit`)
- tray/menu bar integration if practical for current Flutter surface
- lightweight quick actions (`Open`, `Connect`, `Disconnect`, `Quit`) or a documented near-term boundary for them
- single-instance / duplicate-window behavior review and mitigation
- reconnect-safe app resume semantics for the existing controller-backed client shell
- docs/tests for expected desktop behavior

### Out of scope
- updater implementation
- public beta onboarding flow
- notarization / signing hardening
- protocol/runtime capability expansion
- mobile UX work

---

## Work items

### 1. Define desktop lifecycle semantics
- [ ] choose and document app behavior for window close on desktop
- [ ] choose and document app behavior for minimize / reopen / quit
- [ ] decide whether “close hides to tray” is supported now or deferred behind platform constraints
- [ ] expose the chosen semantics in product copy and/or settings if needed

### 2. Introduce desktop lifecycle boundary
- [ ] add a platform service boundary for desktop lifecycle operations
- [ ] keep UI shell separate from platform-specific tray/window code
- [ ] support at least no-op / stub implementations for unsupported platforms so the app model stays testable

### 3. Add tray/menu or equivalent quick-entry surface
- [ ] evaluate a pragmatic Flutter desktop plugin path for tray/menu support
- [ ] wire at least `Open` + `Quit` if full action surface is too risky for first cut
- [ ] if feasible, add `Connect` / `Disconnect` actions against the existing controller boundary
- [ ] show selected profile + current status in label/tooltip when practical

### 4. Reduce duplicate-launch ambiguity
- [ ] define expected behavior when the app is launched twice
- [ ] prefer “focus existing window / existing app instance” semantics if practical
- [ ] if true single-instance behavior is too heavy for this sprint, add an explicit documented limitation and guardrail

### 5. Add regression coverage and docs
- [ ] add tests for lifecycle policy mapping / quick-action gating where testable
- [ ] document known lifecycle limitations by platform
- [ ] update docs index and backlog/progress notes

---

## Suggested delivery order

1. desktop lifecycle policy + docs
2. desktop lifecycle boundary
3. tray/menu first cut (`Open` / `Quit`, then `Connect` / `Disconnect` if stable)
4. duplicate-launch mitigation
5. tests and polish

---

## Risks

### Risk 1 — Platform plugin churn
Desktop tray/window packages may introduce platform-specific instability.

**Mitigation:**
- prefer a thin platform boundary instead of spreading plugin calls across UI
- start with the smallest useful tray/menu slice
- keep fallback/no-op adapters so Linux/macOS/Windows differences do not infect the app model

### Risk 2 — UI lifecycle and runtime lifecycle getting tangled
If tray/window actions talk directly to runtime internals, product code will rot fast.

**Mitigation:**
- route quick actions through the existing controller/application boundary
- keep window lifecycle, app lifecycle, and runtime lifecycle as separate concepts

### Risk 3 — Overreaching into beta-distribution work
Tray/update/validation work can easily blur together and blow scope.

**Mitigation:**
- Sprint 4 is about **desktop behavior predictability**, not full release readiness
- leave updater and public-beta packaging policy to later work

---

## Definition of done

Sprint 4 is done only when:

- a user can explain what happens when they close, minimize, reopen, or quit the app
- the app no longer feels ambiguous as a desktop background companion
- any tray/menu behavior that exists is routed through a clear boundary and documented
- duplicate-launch behavior is either mitigated or explicitly documented as a known limitation
