# v1.3.0 Sprint 2 Plan — Diagnostics and Support UX

## Goal

Make the desktop client supportable enough for beta testing without requiring internal project context.

This sprint executes **Epic C — Diagnostics and support UX** from `docs/v1.3.0-backlog.md`.

---

## Sprint outcome

After this sprint:

- diagnostics export is easier to discover from the main product surfaces
- support bundle export is explained before users export it
- user-facing error categories are more consistent across product surfaces
- troubleshooting content is organized into clearer support-oriented sections
- diagnostics/export failure handling has dedicated regression coverage

---

## Scope

### In scope
- diagnostics export entrypoint visibility
- support bundle summary before export
- support-oriented troubleshooting structure
- user/config/runtime/OS error category presentation
- diagnostics export messaging and regression tests

### Out of scope
- crash persistence / stale session recovery
- tray/menu integration
- update channel work
- broad release validation docs

---

## Work items

### 1. Make diagnostics/support entrypoints more visible

- [ ] audit current entrypoints from Dashboard / Profiles / Advanced
- [ ] add a clearer diagnostics/support entrypoint from the main product surface
- [ ] ensure wording is support-oriented, not implementation-oriented

### 2. Add support bundle summary before export

- [ ] define a short “what this bundle includes / excludes” summary
- [ ] show the summary before export action
- [ ] distinguish preview vs export more clearly in the UI

### 3. Improve troubleshooting structure

- [ ] break troubleshooting into clearer cards/sections
- [ ] surface the top recurring failure categories first
- [ ] make export action live naturally inside the troubleshooting flow

### 4. Improve error category presentation

- [ ] distinguish user/config/runtime/OS error categories consistently
- [ ] align diagnostics/troubleshooting wording with Sprint 1 lifecycle guidance
- [ ] avoid exposing raw internal messages as primary UX copy when a clearer category is available

### 5. Add regression tests

- [ ] diagnostics page widget tests for preview/export CTA visibility
- [ ] tests for support bundle summary visibility
- [ ] tests for export failure messaging
- [ ] tests for troubleshooting structure / primary actions

---

## Suggested delivery order

1. diagnostics/troubleshooting audit
2. support bundle summary
3. main-surface diagnostics entrypoint
4. troubleshooting structure improvements
5. regression tests and polish

---

## Definition of done

Sprint 2 is done only when:

- a tester can find diagnostics export without hunting through the UI
- the app explains what the support bundle contains before export
- troubleshooting feels like a product support surface, not a raw debug dump
- diagnostics/export flows have explicit regression coverage
