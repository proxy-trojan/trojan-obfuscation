# v1.4.0 Release Gates

## Purpose

Define what must be true before `v1.4.0` can be promoted from milestone-complete to releasable.

---

## Gate 1 — Product truth
- [ ] real runtime path exists on supported desktop targets
- [ ] user can distinguish real / simulated / degraded backend state
- [ ] readiness blockers are shown before connect
- [ ] stale-session and retry recovery are understandable

## Gate 2 — Support truth
- [x] support bundle contains real runtime evidence *(substantially met via runtime-aware diagnostics/export wiring and normalized runtimeEvidence payloads)*
- [x] launch/config/environment/connect failures are distinguishable at the family level *(via `FailureFamily` classification flowing through controller persistence, dashboard wording, support guidance, export payload, and diagnostics UI)*
- [x] Advanced/support surfaces do not overclaim certainty *(improved via Diagnostics + Advanced runtime evidence snapshots and family-aware wording)*

### Current snapshot — 2026-04-14
Support truth is now materially stronger than the earlier beta line: the main failure families are no longer collapsed into one generic blob. Remaining risk is not “missing evidence”, but whether future runtime paths keep classification fidelity under more real packaged validation.

## Gate 3 — Release truth
- [x] client asset version metadata matches release tag semantics *(validated by `scripts/validate_client_release_truth.py` across pubspec / packaging metadata / workflow state)*
- [x] product-visible version/channel matches shipped release metadata *(current candidate truth aligned to `1.4.0` stable release metadata)*
- [x] release notes / known limitations reflect actual runtime posture *(current-state docs track the stable `v1.4.0` posture and avoid legacy beta drift)*

### Current snapshot — 2026-04-14
Release truth is no longer just a documentation aspiration. The candidate now has an executable release-truth check that fails when pubspec, packaging metadata, and product-visible workflow state drift apart.

## Gate 4 — Validation truth
- [x] build and packaging are green *(runner-backed: `Client Packaging` run `24404269034`, desktop lanes all success)*
- [x] runtime-aware validation has at least one higher-signal packaged smoke *(client packaging workflow now runs packaged smoke per desktop platform before artifact upload)*
- [x] hard blockers vs advisory validation signals are documented
- [x] release can be justified without hidden tribal knowledge

## Non-gates

The following are valuable, but should not silently redefine `v1.4.0`:
- mobile parity
- updater implementation
- code-signing perfection
- broad public onboarding polish
- protocol expansion beyond current stable backend scope
