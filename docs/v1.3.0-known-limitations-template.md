# v1.3.0 Desktop Beta — Known Limitations

Use this as the starting template for candidate release notes / beta handoff notes.

---

## What this build is

- Desktop-first beta candidate for the Trojan-Pro client shell
- Focused on connection lifecycle clarity, diagnostics/supportability, crash recovery, desktop lifecycle polish, and update-channel boundary definition

## What this build is NOT

- not a production-proven self-updater
- not a fully validated multi-platform installer pipeline
- not a mobile release
- not a guarantee that real runtime integration is equally validated on every desktop platform

## Current limitations

### 1. Updater is stub-only
- update channel is visible in product settings
- update checks currently exercise a local/stub product boundary only
- no remote release feed or signed metadata verification is wired yet

### 2. Packaging maturity varies by platform
- Linux desktop shell and smoke path are ahead of Windows/macOS validation depth
- Windows/macOS still require explicit candidate validation before wider beta claims

### 3. Tray behavior depends on platform support
- some Linux environments may lack appindicator support
- in that case the current build should still launch, but tray behavior degrades to a no-op fallback
- tray visibility/focus behavior can vary by desktop environment

### 4. Duplicate-launch mitigation is best-effort
- secondary launches should focus the primary instance and exit
- focus restoration can still vary across desktop environments/window managers

### 5. Real runtime validation is narrower than UI validation
- widget/unit coverage is strong
- real runtime smoke should still be treated as a separate release gate

## Recommended tester posture

- use Diagnostics export when reporting runtime issues
- treat unexpected update behavior as a documentation/UX issue, not as a supported updater workflow
- report platform-specific tray/focus problems with desktop environment details
