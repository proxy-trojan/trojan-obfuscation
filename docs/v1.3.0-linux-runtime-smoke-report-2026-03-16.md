## Key observations

### 1. Linux no longer hard-fails desktop build when appindicator is missing

Observed fallback behavior:

```text
tray_manager: appindicator dependency not found on Linux; building no-op tray plugin fallback.
```

And at runtime:

```text
DesktopLifecycleService: tray init failed: PlatformException(
  tray_unavailable,
  Linux tray support is unavailable because appindicator is missing.,
  null,
  null
)
```

This means the current Linux posture is now:

- **build succeeds** without appindicator packages
- tray support becomes **unavailable / no-op fallback** in that environment
- app-level lifecycle policy can still degrade to non-tray behavior without crashing the build

### 2. Headless Broadway smoke produced partial runtime evidence, not a full GUI pass

The release bundle was launched under GTK broadway in an attempt to collect minimum runtime evidence on a headless host.

Observed result:

```text
Trojan-Pro Client: another desktop instance is already running; secondary launch exits.
```

So the current release bundle does show evidence that:

- the **single-instance / duplicate-launch guard** is wired into the shipped Linux bundle
- a **secondary launch exits cleanly** instead of opening another desktop instance

But the same headless attempt also hit rendering-surface failures:

```text
Failed to create platform view rendering surface
Unsupported GDK backend, unable to get EGL display
Failed to create OpenGL context: The current backend does not support OpenGL
```

So this run does **not** count as a full GUI validation pass.

## Remaining limitations

The environment used for this smoke run is still effectively headless for Flutter desktop validation:

- no normal `DISPLAY` / `WAYLAND_DISPLAY` desktop session was available
- `xvfb-run` / `Xvfb` were unavailable
- GTK broadway provided only partial evidence and could not validate end-to-end rendering behavior

There were also desktop-environment-specific warnings during the run:

```text
libsecret_error: Failed to unlock the keyring
```

These warnings did not block the packaging/build verdict, but they reinforce that this host is **not** a clean representative desktop validation environment.

## Interpretation

This is no longer a packaging blocker. It is now a **release posture / validation gap**:

- Linux build + test posture is healthy enough for a **staging candidate**
- duplicate-launch behavior has at least **runtime evidence** in the shipped bundle
- Linux GUI/tray behavior still needs validation in a **real desktop session** (or xvfb-capable GUI smoke)

## Release impact

Current honest release posture for `v1.3.0` on Linux:

- ✅ suitable for **small beta / staging handoff**
- ⚠️ not yet strong enough to claim **broad public desktop beta confidence**

Before promoting `v1.3.0` as a wider desktop beta candidate, the remaining Linux-specific confidence gap is:

1. validate GUI launch in a real desktop or xvfb-capable environment
2. explicitly document that missing appindicator means **no tray**, not **no build**
3. verify one real release-bundle smoke pass for window/tray/lifecycle behavior end-to-end
