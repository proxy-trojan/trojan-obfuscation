## Key observation

Linux no longer hard-fails desktop build when appindicator is missing.

Observed fallback behavior:

```text
tray_manager: appindicator dependency not found on Linux; building no-op tray plugin fallback.
```

This means the current Linux posture is now:

- **build succeeds** without appindicator packages
- tray support becomes **unavailable / no-op fallback** in that environment
- app-level lifecycle policy can still degrade to non-tray behavior

## Remaining limitation

The environment used for this smoke run was headless:

```text
GUI launch: skipped (no DISPLAY / WAYLAND_DISPLAY and xvfb-run unavailable)
```

So this run does **not** count as a full GUI validation pass.

## Interpretation

This is no longer a hard packaging blocker. It is now a **release posture / validation gap**:

- Linux headless smoke is healthy enough to build and preflight
- Linux GUI/tray behavior still needs validation in a real desktop session (or xvfb-backed GUI smoke)

## Release impact

Before promoting `v1.3.0` as a desktop beta candidate, the remaining Linux-specific confidence gap is:

1. validate GUI launch in a real desktop or xvfb-capable environment
2. explicitly document that missing appindicator means **no tray**, not **no build**
3. decide whether this fallback is acceptable for beta release posture

## Next recommended action

- keep the Linux no-op tray fallback
- document tray fallback behavior in beta notes / lifecycle docs
- run one GUI-capable Linux smoke pass to confirm window/tray/lifecycle behavior end-to-end
